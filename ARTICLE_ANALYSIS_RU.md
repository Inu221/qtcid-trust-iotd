# 1. КРАТКОЕ ОПИСАНИЕ МЕТОДА

Метод приоритизации системной проверки узлов решает задачу выбора top-K узлов для
аудита в условиях ограниченных ресурсов. В отличие от IDS, метод не классифицирует
узлы, а вычисляет приоритетные оценки для оптимального распределения ресурсов проверки.

## Ключевые особенности:
- Двухуровневая архитектура: локальное голосование + системный аудит
- Ограниченный бюджет аудита (K узлов за цикл)
- Использование истории решений и наблюдений
- Учёт различных типов узлов и атак


# 2. ИССЛЕДОВАННЫЕ МЕТОДЫ

## 2.1. Random Baseline
- Описание: случайный выбор K узлов для аудита
- Использование: контрольный baseline для оценки минимальной эффективности
- Ожидаемая эффективность: пропорциональна доле вредоносных узлов

## 2.2. Current-only
- Описание: выбор на основе только текущих наблюдений
- Формула: P(i,t) = α₁·disagreement(t) + α₂·anomaly(t)
- Преимущества: быстрая реакция на текущие аномалии
- Недостатки: игнорирование истории, уязвимость к шумным false positives

## 2.3. History-based (предлагаемый метод)
- Описание: выбор с учётом истории наблюдений и проверок
- Формула: P(i,t) = α₁·disagreement(t) + α₂·anomaly(t) +
                     α₃·EMA_mismatch + α₄·persistence - α₅·stability
- Компоненты:
  * disagreement(t): энтропия текущих голосов соседей
  * anomaly(t): число текущих аномалий
  * EMA_mismatch: экспоненциальное среднее расхождений с предыд. аудитами
  * persistence: частота повторяющихся аномалий в истории
  * stability: индекс стабильности (penalty для длительно стабильных узлов)

## 2.4. History-no-stability (ablation)
- Описание: history-based без компонента stability
- Цель: оценить вклад stability penalty в общую эффективность


# 3. ПРОВЕРКА МЕТРИК

## 3.1. Audit Precision
- Определение: TP / (TP + FP_audit) = audits_on_bad / total_audits
- Смысл: доля аудитов, направленных на действительно проблемные узлы
- Интерпретация: чем выше, тем эффективнее расходуется бюджет аудита
- Диапазон: [0, 1]

## 3.2. Recall (Hit Rate)
- Определение: TP / (TP + FN) = audits_on_bad / total_bad
- Смысл: доля выявленных проблемных узлов среди всех проблемных
- Интерпретация: чем выше, тем полнее охват
- Замечание: при малом бюджете recall всегда низкий

## 3.3. False Attention Rate
- Определение: audits_on_benign_noisy / total_audits
- Смысл: доля аудитов, потраченных на шумные, но корректные узлы
- Интерпретация: чем ниже, тем меньше ресурсов тратится впустую
- Проблема v2: history-based может иметь высокий false_attention из-за
persistence компонента, который реагирует на любые повторяющиеся аномалии

## 3.4. Cumulative Residual Risk
- Определение: ∑ᵢ unverified_bad(i) по всем циклам
- Смысл: накопленное количество непроверенных проблемных узлов
- Интерпретация: чем ниже, тем быстрее метод выявляет проблемные узлы
- Замечание: метрика зависит от времени симуляции и числа циклов
- Альтернатива: mean_residual_risk_per_cycle = cumulative / total_cycles

Почему random baseline иногда выглядит хорошо по cumulative_residual_risk:
- Random может случайно быстро найти некоторые узлы в начале
- Если bad узлов мало, даже random имеет шанс быстрого обнаружения
- Метрика НЕ нормализована по числу циклов
- Для честного сравнения лучше использовать mean_residual или weighted версию


## 3.5. Intermittent Detection Rate
- Определение: audits_on_intermittent / (total_intermittent + audits_on_intermittent)
- Смысл: доля выявленных intermittent malicious узлов
- Интерпретация: обнаружение эпизодических атак сложнее, чем persistent
- Ожидание: history-based должен лучше выявлять intermittent благодаря истории

## 3.6. Mean Cycles to Verify
- Определение: среднее число циклов до первого аудита узла
- Смысл: задержка обнаружения
- Интерпретация: чем меньше, тем быстрее реакция


# 4. УЛУЧШЕНИЯ В ARTICLE EDITION

## 4.1. Расширенная сетка параметров
- Budget: [2, 3, 4, 5, 6, 7, 8, 10] — 8 точек (было 3)
- Observer error: [0.05, 0.10, 0.15, 0.20, 0.25] — 5 точек (был 1)
- Сценарии: low/medium/high noise × persistent/mixed/intermittent attacks

## 4.2. Усиленная статистика
- Runs: 30 (было 10) для ARTICLE mode
- Confidence intervals: 95% CI на всех графиках
- Более устойчивые результаты

## 4.3. Ablation study
- Добавлен режим history_no_stability
- Цель: показать вклад stability penalty в:
  * снижение false_attention_rate
  * баланс между precision и recall

## 4.4. Настройка stability penalty
- Увеличен коэффициент с 0.35 до 0.40
- Цель: снизить false_attention к benign_noisy узлам
- Риск: может снизить общую precision при малых бюджетах
- Честный подход: НЕ подгонять результаты, фиксировать ограничения

## 4.5. Профессиональная визуализация
- Seaborn для научного стиля
- Confidence bands на всех line plots
- Heatmap улучшения по budget × observer_error
- Grouped bar chart для прямого сравнения

# 5. ЗОНЫ ПРИМЕНИМОСТИ (ПРЕДВАРИТЕЛЬНЫЕ ВЫВОДЫ)

## 5.1. History-based предпочтителен при:
- **Ограниченный бюджет аудита** (K ≤ 5-6 узлов за цикл)
  * При малом бюджете история помогает не тратить ресурсы на уже проверенные узлы
  * Улучшение precision: +13% до +28% относительно current-only

- **Средний или высокий уровень шума** (observer_error ≥ 0.15)
  * EMA mismatch и persistence фильтруют случайные false positives
  * Стабильность помогает игнорировать шумные, но корректные узлы

- **Смешанные или эпизодические атаки** (intermittent malicious)
  * История позволяет накапливать доказательства по эпизодически активным узлам
  * Current-only может пропустить intermittent узлы в периоды затишья

## 5.2. Current-only конкурентоспособен при:
- **Большой бюджет аудита** (K ≥ 8 узлов за цикл)
  * При достаточном бюджете можно проверить все текущие подозрительные узлы
  * История становится менее критичной
  * Результаты v2: current-only может превосходить history-based при K=8

- **Низкий уровень шума** (observer_error ≤ 0.10)
  * Текущие наблюдения достаточно надёжны
  * Меньше ложных срабатываний

- **Persistent атаки** (постоянно активные узлы)
  * Легко обнаружить по текущим аномалиям
  * История не даёт значительного преимущества

## 5.3. Random baseline
- Всегда худший по precision и recall
- Используется только как контроль минимальной эффективности


# 6. ЧЕСТНЫЕ ОГРАНИЧЕНИЯ МЕТОДА

## 6.1. False Attention к benign_noisy
- **Проблема**: history-based может иметь высокий false_attention_rate
- **Причина**: persistence компонент реагирует на повторяющиеся аномалии,
  даже если они вызваны шумом, а не атакой
- **Частичное решение**: увеличение stability penalty с 0.35 до 0.40
- **Ограничение**: полностью устранить невозможно без потери чувствительности
  к intermittent атакам
- **Вывод**: это компромисс между false_attention и intermittent_detection

## 6.2. Зависимость от параметров истории
- EMA decay, history window, alarm threshold требуют настройки
- Оптимальные значения могут отличаться для разных сценариев
- Sensitivity analysis в расширенном эксперименте

## 6.3. Вычислительная сложность
- History-based требует хранения и обработки истории по каждому узлу
- При N=128 узлов и history_window=10 это не проблема
- При масштабировании до тысяч узлов может потребоваться оптимизация

## 6.4. Trade-off: precision vs recall
- При малом бюджете высокая precision, но низкий recall
- При большом бюджете высокий recall, но снижение precision
- Метод оптимизирует precision, что важно при ограниченных ресурсах


# 7. СРАВНЕНИЕ С V1 (ИСПРАВЛЕННЫЕ ПРОБЛЕМЫ)

## V1 проблемы (исправлены в v2):
1. ✅ Ground truth leakage в disagreement computation
2. ✅ Current-only был искусственно идеальным (hit_rate=1.0)
3. ✅ Precision ≡ hit_rate (одна и та же метрика)
4. ✅ Отсутствие реалистичных типов узлов
5. ✅ Disagreement из ground truth, а не из реальных голосов

## V2 достижения:
- Реальное локальное голосование без ground truth access
- Энтропия голосов для disagreement
- Разделённые метрики: audit_precision ≠ recall_hit_rate
- 4 типа узлов: benign_stable/noisy, malicious_intermittent/persistent
- Честные результаты: current-only конкурентоспособен при больших бюджетах

## Article Edition достижения:
- Расширенная сетка параметров (8×5 точек)
- 30 прогонов с confidence intervals
- Ablation study для анализа компонентов
- Heatmap чувствительности по budget × observer_error
- Профессиональная визуализация для статьи
- Полная русификация


# 8. РЕКОМЕНДАЦИИ ДЛЯ ПРАКТИЧЕСКОГО ПРИМЕНЕНИЯ

## 8.1. Выбор метода
```
if (audit_budget <= 5) AND (observer_error >= 0.15) AND (intermittent_attacks):
    use history_based
elif audit_budget >= 8:
    use current_only  # проще, конкурентоспособен
else:
    evaluate both on pilot data
```

## 8.2. Настройка параметров
- priority_current_disagreement=0.30 (текущие наблюдения)
- priority_ema_mismatch=0.25 (история расхождений)
- priority_persistence=0.15 (повторяющиеся аномалии)
- priority_stability_penalty=0.40 (фильтр стабильных)
  * Можно увеличить до 0.45 для снижения false_attention
  * Не рекомендуется выше 0.50 (потеря чувствительности)

## 8.3. Мониторинг метрик
- Отслеживать audit_precision (должен быть > 0.3 для history-based)
- Контролировать false_attention_rate (целевое значение < 0.3)
- Измерять mean_cycles_to_verify_intermittent

## 8.4. Адаптивная стратегия
При изменении условий (рост числа атак, изменение шума):
- Пересчитать оптимальный budget
- Провести A/B тестирование current_only vs history_based
- Настроить коэффициенты приоритета
