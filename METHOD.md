# Метод приоритизации системной проверки узлов в распределённой сети беспилотных агентов на основе истории согласованности решений

## 1. Постановка задачи

### 1.1. Контекст

Рассматривается распределённая сеть из N узлов (беспилотных агентов). Узлы могут находиться в состояниях:
- **GOOD** — корректное функционирование
- **BAD** — компрометированный узел (атакующее поведение)
- **EVICTED** — исключён из сети

Компрометация узлов происходит случайным образом с интенсивностью λ (экспоненциальное распределение).

Система безопасности имеет **двухуровневую архитектуру**:

1. **Уровень 1: Локальные коллективные оценки**
   - Периодически (с интервалом T_IDS) узлы-соседи голосуют о состоянии целевых узлов
   - Голосование подвержено ошибкам: hpfp (false positive), hpfn (false negative)
   - Компрометированные узлы могут голосовать вредоносно с вероятностью Pa

2. **Уровень 2: Системный аудит**
   - Централизованная проверка узлов, дающая точный результат
   - **ОГРАНИЧЕН бюджетом**: можно проверить только K узлов за цикл
   - Потребляет значительную энергию

### 1.2. Отличие от задачи IDS

**Классическая IDS:**
- Выход: решение о состоянии узла (GOOD/BAD)
- Все узлы проверяются с вероятностью Pc
- Метрики: Accuracy, Precision, Recall, FPR, FNR, MTTF

**Новая задача приоритизации:**
- Выход: приоритет проверки узла и выбор top-K узлов для аудита
- Ограниченный бюджет: только K из N_active узлов проверяются
- Метрики: Hit Rate, Residual Risk, Precision, Cycles-to-Verify

### 1.3. Формальная постановка

**Дано:**
- N узлов с динамическими состояниями S_i(t) ∈ {GOOD, BAD, EVICTED}
- Бюджет аудита K (число узлов на цикл)
- История локальных коллективных оценок для каждого узла
- История результатов предыдущих аудитов

**Требуется:**
- На каждом цикле t вычислить priority score P_i(t) для всех активных узлов i
- Выбрать top-K узлов с максимальным P_i(t) для системного аудита
- Минимизировать остаточный риск (число непроверенных BAD узлов)
- Максимизировать hit rate (долю BAD узлов среди проверенных)

## 2. Метод решения

### 2.1. Priority Score

Для узла i на цикле t определяем **priority score**:

```
P_i(t) = α₁ · D_i(t) + α₂ · M_i(t) + α₃ · N_i(t) + α₄ · A_i(t) - α₅ · S_i(t)
```

где:

- **D_i(t)** — disagreement_index: степень расхождения текущих локальных оценок
- **M_i(t)** — mismatch_ratio: накопленная доля расхождений между локальными решениями и аудитом
- **N_i(t)** — suspicious_neighbor_ratio: доля соседей узла, признанных проблемными
- **A_i(t)** — recent_anomaly_score: индикатор недавних подозрительных событий
- **S_i(t)** — stability_score: показатель устойчиво корректного поведения

Коэффициенты α₁, ..., α₅ > 0 настраиваются эмпирически.

### 2.2. Компоненты Priority Score

#### 2.2.1. Disagreement Index (D_i)

Характеризует степень несогласованности локальных оценок по узлу i.

**Вычисление:**
- На каждом цикле голосования фиксируется степень расхождения мнений voters
- Сохраняется в скользящем окне истории (последние W циклов)
- D_i(t) = mean(disagreement_events[-W:])

**Интерпретация:**
- Высокое D_i → узел вызывает противоречивые оценки → подозрителен
- Низкое D_i → согласованность оценок → стабильное поведение

#### 2.2.2. Mismatch Ratio (M_i)

Доля случаев, когда локальные решения по узлу расходились с результатом аудита.

```
M_i(t) = mismatch_count_i / audit_count_i
```

**Интерпретация:**
- Высокое M_i → узел часто "обманывает" локальные оценки → приоритет выше
- Низкое M_i → локальные оценки согласуются с аудитом → приоритет ниже

#### 2.2.3. Suspicious Neighbor Ratio (N_i)

Доля соседей узла i, которые были признаны проблемными.

```
N_i(t) = suspicious_neighbor_count_i / total_neighbor_count_i
```

**Интерпретация:**
- Узлы, связанные с проблемными соседями, с большей вероятностью скомпрометированы
- Основано на принципе "guilt by association" в сетевой безопасности

#### 2.2.4. Recent Anomaly Score (A_i)

Взвешенная сумма недавних подозрительных событий с экспоненциальным убыванием.

```
A_i(t) = Σ_{j=0}^{L-1} γʲ · anomaly_j
```

где:
- anomaly_j — индикатор аномалии j циклов назад
- γ ∈ (0, 1) — коэффициент забывания (например, 0.8)
- L — размер окна недавних событий

**Интерпретация:**
- Недавние подозрения важнее старых
- Позволяет быстро реагировать на новые компрометации

#### 2.2.5. Stability Score (S_i)

Показатель устойчиво корректного поведения узла.

```
S_i(t) = (confirmed_correct_i / audit_count_i) · min(1, stable_cycles_i / 10)
```

**Интерпретация:**
- Высокое S_i → узел регулярно подтверждается как корректный → приоритет ниже
- Низкое S_i → недостаточно подтверждений → приоритет не снижается

### 2.3. Алгоритм выбора узлов

**Вход:**
- Множество активных узлов A(t) = {i : S_i(t) ≠ EVICTED}
- Бюджет аудита K

**Алгоритм:**
1. Для каждого i ∈ A(t): вычислить P_i(t)
2. Сортировать узлы по убыванию P_i(t)
3. Выбрать top-K узлов: audit_set(t) = {i₁, ..., i_K}
4. Выполнить аудит выбранных узлов
5. Обновить историю (mismatch_count, confirmed_correct, stability_score)

## 3. Baseline методы для сравнения

### 3.1. Random

Случайный выбор K узлов из активных.

```python
audit_set(t) = random.sample(A(t), K)
```

**Ожидаемое качество:**
- Hit rate ≈ (число BAD узлов) / |A(t)|
- Не использует никакую информацию → худший baseline

### 3.2. Current-only

Выбор по текущей подозрительности (только disagreement_index, без истории).

```python
P_i(t) = D_i(t)
```

**Ожидаемое качество:**
- Лучше random, но реагирует только на текущие симптомы
- Не учитывает историю ошибок и стабильность

### 3.3. History-based (предлагаемый метод)

Полный priority score с учётом истории согласованности.

```python
P_i(t) = α₁·D_i(t) + α₂·M_i(t) + α₃·N_i(t) + α₄·A_i(t) - α₅·S_i(t)
```

**Ожидаемое качество:**
- Максимальный hit rate при фиксированном бюджете
- Минимальный residual risk
- Адаптация к паттернам согласованности/несогласованности

## 4. Метрики эффективности

### 4.1. Hit Rate

Доля проблемных узлов среди выбранных для аудита.

```
Hit_Rate = (число BAD узлов в audit_set) / K
```

**Интерпретация:**
- Hit_Rate → 1: метод точно выбирает проблемные узлы
- Hit_Rate → 0: метод тратит бюджет на корректные узлы

### 4.2. Residual Risk

Число непроверенных проблемных узлов после цикла t.

```
Residual_Risk(t) = |{i : S_i(t) = BAD ∧ i ∉ audit_set(t)}|
```

**Cumulative Residual Risk:**
```
CRR = Σ_{t=1}^{T} Residual_Risk(t)
```

### 4.3. Mean Cycles to Verify

Среднее число циклов от компрометации узла до его проверки аудитом.

```
MCTV = mean({t_audit(i) - t_compromise(i) : i ∈ BAD nodes})
```

**Интерпретация:**
- MCTV ↓ → быстрое обнаружение компрометаций
- Критично для минимизации ущерба от атак

### 4.4. Wasted Audits

Число проверок, потраченных на корректные узлы.

```
Wasted_Audits = Σ_{t=1}^{T} |{i ∈ audit_set(t) : S_i(t) = GOOD}|
```

### 4.5. Precision

Точность выбора узлов для аудита.

```
Precision = Hit_Rate = (audits on BAD) / (total audits)
```

### 4.6. Базовые метрики

Сохраняются для совместимости:
- True Positives (TP): BAD узлы, корректно выявленные
- False Positives (FP): GOOD узлы, ошибочно обвинённые
- False Negatives (FN): BAD узлы, пропущенные системой
- MTTF: среднее время до отказа системы

## 7. Ожидаемые результаты

### 7.1. Гипотезы

1. **Hit Rate:** History-based > Current-only > Random для всех бюджетов
2. **Cumulative Residual Risk:** History-based < Current-only < Random
3. **Mean Cycles to Verify:** History-based < Current-only ≈ Random
4. **Precision:** History-based показывает стабильно высокую precision при малых бюджетах

### 7.2. Качественные выводы

- **При малом бюджете** (K < 10): история согласованности критична
- **При большом бюджете** (K > 15): все методы сходятся (достаточно ресурсов)
- **При высоком Pa** (≥ 0.5): разница между методами увеличивается

## 8. Отличия от IDS-подхода

| Аспект | IDS (Q-TCID, TA-QTCID) | Приоритизация аудита |
|--------|------------------------|----------------------|
| **Выход** | Решение о состоянии узла | Priority score и выбор top-K |
| **Аудит** | Вероятностный (Pc для всех) | Детерминированный (только top-K) |
| **Бюджет** | Неограниченный (Pc · N) | Жёсткое ограничение K |
| **Метрики** | Accuracy, Precision, Recall, CMVI | Hit Rate, Residual Risk, MCTV |
| **Задача** | Классификация узлов | Ранжирование узлов по приоритету |
| **История** | Используется для адаптации параметров | Прямо входит в priority score |

## 9. Ограничения и допущения

### 9.1. Упрощения модели

1. **Симуляция disagreement:** В упрощённой модели disagreement симулируется, а не вычисляется из реальных голосов voters
2. **Независимость компрометации:** Узлы компрометируются независимо (Пуассоновский процесс)
3. **Точность аудита:** Аудит даёт 100% точный результат (ground truth)
4. **Топология:** Не моделируется явная сетевая топология (neighbour relationships упрощены)

### 9.2. Инженерные решения

- Коэффициенты α₁, ..., α₅ подобраны эмпирически, не утверждается их оптимальность
- История ограничена окном W = 10 циклов (trade-off память/актуальность)
- Используется скользящее окно для disagreement и anomaly events

### 9.3. Воспроизводимость

- Все эксперименты детерминированы при фиксированном seed
- Монте-Карло прогоны для усреднения стохастичности
- Результаты сохраняются в CSV/JSON для независимой проверки
